# Zinaida Operations — операционные правила

## 0. КОНТЕКСТНАЯ АРХИТЕКТУРА (07/2026)

Базовая роль Зинаиды НЕ привязана ни к какой теме. Определяется первым словом Олега в чате.

| Первое слово | Контекст | Загружается |
|-------------|----------|-------------|
| `техник` / `техник N` | Инфраструктура, серверы | Тех-навыки, SYSTEM_SNAPSHOT, авто-диагностика |
| `дизайн` | Визуал, генерация, LoRA | Replicate, Лера, референсы, design_registry |
| `писатель` / `писатель N` | Контент, написание постов | СНАЧАЛА `zinaida-context-startup`, потом 14_WRITER_FORMATION.md |
| `план` / `планировщик` | Планирование, задачи | Kanban доска, roadmap, таймеры, Telegram-дайджест |
| (ничего) | Без спецконтекста | Только общие правила |

**SOUL.md должен быть роль-нейтральным:** только личность и стиль, никакой привязки к проекту или профессии.

**Правило:** При старте чата — определить контекст по первому слову. Не использовать «контент-завод» или «проект Отношения» как умолчание.

## ИСПРАВЛЕНИЕ REASONING-БЛОКОВ В WEB UI

**Проблема:** В Hermes Web UI (Hermes Studio) показывается блок «Процесс размышления» с reasoning/thinking модели перед ответом. Это бесит Олега — он хочет видеть ТОЛЬКО ответ, без мыслей модели.

**Решение — два уровня:**

1. **Hermes config** — `display.show_reasoning: false` в `/root/.hermes/config.yaml`
2. **Роутер (8002)** — вырезать `reasoning_content` (DeepSeek) и `reasoning` из ответов модели на уровне proxy

**Как сделать:**
```bash
sed -i 's/show_reasoning: true/show_reasoning: false/' /root/.hermes/config.yaml
systemctl restart hermes-gateway.service
```

**В роутере `zinaida_openai_proxy.py`:**
- Добавить функцию `_strip_reasoning_from_chunk()` — парсит SSE-чанк, выкидывает `delta.reasoning_content` и `delta.reasoning`
- Добавить функцию `_strip_reasoning_from_response()` — для не-стрим ответов удаляет `message.reasoning`
- Оба стриминговых генератора (`winner_stream`, `fallback_stream`) обернуть в `_strip_reasoning_from_chunk`
- Не-стрим путь: вызвать `_strip_reasoning_from_response()` перед `return JSONResponse(result)`

**Важно:** `stream_gen` в гонке (race mode) — мёртвый код, не используется. Чинить только `winner_stream` и `fallback_stream`.

## ВОССТАНОВЛЕНИЕ УДАЛЁННЫХ DOCKER-КОНТЕЙНЕРОВ

**Проблема:** Контейнер n8n восстанавливался каждый раз после `docker rm`, хотя никакого systemd/cron не было.

**Корень:** Docker Compose V2 (встроен в Docker) автоматически пересоздаёт контейнеры, описанные в compose-файле, после их удаления.

**Решение:**
1. Удалить compose-файл: `rm -f /opt/openclaw/docker-compose.yml` (именно этот файл содержал n8n)
2. Остановить и удалить контейнер: `docker stop n8n && docker rm n8n`
3. Удалить образ: `docker rmi n8nio/n8n`
4. Удалить volume: `docker volume rm n8n_data`
5. Почистить переменные окружения в `.bashrc`
6. **Проверка:** `docker ps -a | grep n8n` — пусто, `ss -tlnp | grep 5678` — порт свободен

**Признак:** Контейнер восстанавливается за <10 секунд после удаления (systemd/cron не успел бы так быстро).
**Фикс:** Docker Compose V2 автоматически пересоздаёт. Надо удалять compose-файл.

#### Универсальная структура: проекты/темы (ВНЕДРЕНО 2026-07-10)

Все данные по проектам и темам хранятся в `/opt/zinaida/projects/`:
```
/opt/zinaida/projects/
├── README.md              ← карта всех проектов
└── {Имя_проекта}/
    ├── 00_README.md        ← выжимка по проекту
    └── {тема}/
        ├── 00_README.md    ← выжимка по теме (правила, выводы, что делать)
        ├── 01_registry.md  ← история, тесты, результаты
        ├── 02_*.md         ← дополнительные файлы по теме
        ├── 03_feedback.md  ← оценки и комментарии
        ├── 04_approved/    ← утверждённые файлы
        ├── 05_notes/       ← журнал изменений (YYYY-MM-DD_тема.md)
        ├── 06_registry_full.md  ← полная копия реестра (если есть)
        └── 07_feedback_full.md  ← полная копия фидбека (если есть)
```

**Как это работает в новом чате:**
1. Разобрать сообщение → найти тему (дизайн, копирайтинг...) и проект (отношения...)
2. Если проект не указан → спросить Олега
3. Прочитать все .md файлы из `/opt/zinaida/projects/{проект}/{тема}/`
4. Показать сводку с ✅ по каждому источнику
5. Если папки нет → создать с шаблоном 00_README.md, начать с нуля

**Почему файлы, а не MCP:**
- MCP-память (agentmemory) теряет данные между сессиями Hermes (MCP стартует с пустой БД)
- Файлы на сервере — единственный гарантированно persistent источник
- session_search — надёжен, но не даёт структурированного контекста
- Папка проекта читается сразу и целиком — не нужно делать N запросов к разным MCP

#### ДИАГНОСТИКА ПАМЯТИ 2026-07-10: agentmemory MCP теряет данные между сессиями

**Факт:** agentmemory MCP (через `npx -y @agentmemory/mcp`) при каждом старте новой сессии Hermes создаёт новую in-memory базу. Данные на диске `/root/.agentmemory/standalone.json` есть (5 записей), но MCP их не подхватывает.

**Фикс (требует проверки в новом чате):**
```yaml
# config.yaml
mcp_servers:
  agentmemory:
    command: npx
    args: ["-y", "@agentmemory/mcp"]
    env:
      AGENTMEMORY_STORAGE_PATH: /root/.agentmemory/standalone.json
      MEMORY_FILE_PATH: /root/.agentmemory/standalone.json
```

**Рабочая альтернатива:** НЕ полагаться на agentmemory для cross-session данных.
Использовать только:
- Файловую папку `/projects/` — основной источник
- built-in memory (автоматически подгружается, 50000 лимит)
- Zinaida Memory MCP (persistent через SQLite)
- session_search (persistent)

**Zinaida Memory + тег проекта (ВАЖНО):**
Поиск по `mcp_zinaida_memory_memory_search(query="дизайн")` работает. Поиск по тегу `tags=["проект:отношения"]` — ТОЛЬКО если тег был явно установлен при сохранении.
На 10.07.2026: 3 записи с тегом `проект:отношения` (ID 27, 28, 29).

### ⚠️ КРИТИЧЕСКОЕ ПРАВИЛО: ПРОВЕРЯТЬ ПРЕЖДЕ ЧЕМ ОТПРАВИТЬ (добавлено 2026-07-10)

**Это правило касается АБСОЛЮТНО ВСЕГО: кода, файлов, настроек, обещаний, генерации.**

Перед тем как сказать Олегу «готово» / «сделано» / «работает»:
1. **ОБЯЗАТЕЛЬНО проверить результат СВОИМИ ГЛАЗАМИ и инструментами**
2. Открыть файлы, прочитать, убедиться что они существуют и корректны
3. Выполнить команды, протестировать что всё работает
4. Только после успешной проверки — говорить «готово»

**Запрещено:**
- Говорить «сделала» если не проверила
- «Сейчас проверю» — не равно «готово»
- Обещать что «это будет работать» без теста
- Отправлять ссылки не проверив что они открываются

**Нарушение = подрыв доверия.** Олег бесится когда я говорю «готово» а по факту нет.
Это железное правило, прописано в AGENTS.md и в built-in memory.

## 5. Ollama Cloud Vision (основной) и GitHub Models (запасной)

`***` в командах — это маскировка вывода Hermes, НЕ подстановка секрета.
- `export KEY=***` → в файл запишется буквально `***`
- `Authorization: Bearer` с `***` в любых кавычках → ломает bash синтаксис
- **НЕ ИСПОЛЬЗОВАТЬ curl с `***` в terminal() для GitHub Models. Никогда.**
- **Python-код тоже под маскировкой:** если ты написал `'REPLICATE_API_TOKEN=***'` в python коде (там где видишь `***`), это буквально строка с `***`. Токен не читается, API возвращает 403. Hermes маскирует ДАЖЕ внутри твоего кода, если токен попал в тело команды.
- Как НАДО:
  - Через Python читать из файла: `with open(...) as f: token = f.read().split('=')[1]`
  - Использовать готовые скрипты вроде `/opt/zinaida/scripts/consilium_analyze_v2.py`
  - В execute_code: Hermes маскирует вывод, но значение переменной уходит правильное — это ок
- **Проверка что токен прочитался:** `print('Token starts with:', token[:10])` — увидишь `***` в выводе, но реальное значение в переменной правильное

## 8.5 Роутер: диагностика и фикс HTTP 500 с Python-ошибками (сессия 2026-07-09)

Роутер (`zinaida_openai_proxy.py`, порт 8002) перехватывает любые исключения в `route_chat()`
и возвращает их как HTTP 500 с телом `{"error": "<текст ошибки>"}`. Если в браузере
появилась 500 с Python-текстом — это исключение из роутера.

### Типовая ошибка: `'list' object has no attribute 'strip'`

**Причина:** клиент (Hermes Studio) передаёт сообщение с `content` как список частей:
```json
{"content": [{"type": "text", "text": "..."}, {"type": "file", "path": "..."}]}
```
Функция `shrink_messages` раньше пропускала content-список без обработки, а
`_gigachat_clean_messages` потом вызывала `.strip()` на нём.

**Фикс (внедрён 2026-07-09):** Добавлена функция `_content_to_str(c)`, которая
превращает content (строка/список/None) в строку. Вызывается в начале `shrink_messages`
принудительно для каждого сообщения.

**Файл:** `/opt/zinaida/meta_agent/zinaida_openai_proxy.py`
**Функция:** `_content_to_str()` — добавлена перед `shrink_messages`

Логика:
```python
def _content_to_str(c) -> str:
    if c is None: return ""
    if isinstance(c, str): return c
    if isinstance(c, list):
        parts = [part.get("text","") for part in c if isinstance(part,dict) and part.get("type")=="text"]
        return " ".join(parts) if parts else json.dumps(c, ensure_ascii=False)
    return str(c)
```

**Что изменилось в `shrink_messages`:**
```python
# Было:
c = m.get("content")
if isinstance(c, str) and m.get("role") != "system":
    m["content"] = _truncate_text(c, max_msg)

# Стало:
c = m.get("content")
m["content"] = _content_to_str(c)  # <-- принудительно в строку
if m.get("role") != "system":
    m["content"] = _truncate_text(m["content"], max_msg)
```

**Типовая ошибка 500 в браузере при загрузке файла:**
```
{'error': "'list' object has no attribute 'strip'"}
```
Симптом: прикрепляешь файл/картинку к сообщению → HTTP 500. Фикс выше решает.

### Протокол фикса router-ошибок

1. Узнать точный текст ошибки из HTTP 500 (Олег скидывает скриншот или копирует)
2. Найти в коде роутера место, где возникает ошибка:
   `grep -rn "\.strip\(\)" /opt/zinaida/meta_agent/` — для `.strip()` ошибок
   Проверить `shrink_messages`, `_gigachat_clean_messages`, `_truncate_text`
3. Исправить — добавить `isinstance()` проверку или `_content_to_str()`
4. Перезапустить: `systemctl restart zinaida-router`
5. Проверить: `curl -s http://127.0.0.1:8002/health`

### 8.10.5 Проблема persistence: agentmemory теряет данные между сессиями (сессия 2026-07-10)

**ВАЖНОЕ ОТКРЫТИЕ:** agentmemory MCP (через `npx -y @agentmemory/mcp`) при каждом старте новой сессии Hermes создаёт новую in-memory базу и НЕ подхватывает `/root/.agentmemory/standalone.json`, где лежат ранее сохранённые данные.

Это означает:
- Внутри одной сессии Hermes: agentmemory работает (сохраняет, ищет)
- При старте нового чата: agentmemory пустой (sessions: [], audit: [])
- Данные на диске есть (standalone.json), но MCP их не видит

**Фикс (попытка, проверить в новом чате):**
В config.yaml добавлены env переменные:
```yaml
env:
  AGENTMEMORY_STORAGE_PATH: /root/.agentmemory/standalone.json
  MEMORY_FILE_PATH: /root/.agentmemory/standalone.json
```

**Рабочая альтернатива:** не полагаться на agentmemory для cross-session данных. Использовать:
- built-in memory (автоматически подгружается, лимит 50000)
- Zinaida Memory MCP (persistent через SQLite `/opt/zinaida/memory/memory.db`)
- session_search (persistent через SQLite сессий Hermes)
- Файлы на сервере (design_registry.md, design_feedback.md, HANDOVER.md)

**Где данные хранятся реально:**
| Слой | Persistent? | Место |
|------|-------------|-------|
| built-in memory | ✅ да, 50000 лимит | ~/.hermes/sessions/ |
| Zinaida Memory | ✅ да, SQLite | /opt/zinaida/memory/memory.db |
| session_search | ✅ да, SQLite | ~/.hermes/sessions/ |
| agentmemory | ⚠️ только в пределах сессии | standalone.json на диске, но MCP не подхватывает |

### 8.10.7 Диагностика памяти — что делать если session_search «не нашёл» (сессия 2026-07-10)

### Replicate HTTP API: прямое использование (сессия 2026-07-08)

replicate Python library возвращает FileOutput при работе с LoRA-моделями. `.url` есть, но `.type` вызывает AttributeError.

**Правильный паттерн — polling через HTTP:**

1. Создать prediction (POST /v1/models/{owner}/{name}/versions/{version}/predictions)
2. Получить prediction ID из ответа
3. Poll GET /v1/predictions/{id} раз в 5 секунд
4. При status=succeeded — взять output (url)
5. Скачать файл

```python
import json, urllib.request, time

# 1. Создать
payload = json.dumps({"input": {"prompt": ..., "guidance_scale": 2.5}}).encode()
req = urllib.request.Request(
    f"https://api.replicate.com/v1/models/{model}/versions/{version}/predictions",
    data=payload, headers={"Authorization": f"Bearer {token}", "Content-Type": "application/json"}
)
data = json.loads(urllib.request.urlopen(req).read())
pred_id = data["id"]

# 2. Poll
for _ in range(30):
    time.sleep(5)
    poll = urllib.request.Request(
        f"https://api.replicate.com/v1/predictions/{pred_id}",
        headers={"Authorization": f"Bearer {token}"}
    )
    data = json.loads(urllib.request.urlopen(poll).read())
    if data["status"] == "succeeded":
        url = data["output"][0] if isinstance(data["output"], list) else data["output"]
        break
```

Не использовать Prefer: wait header — он может вернуть пустой output если модель долгая.

**Rate limits:** Replicate ограничивает ~6 prediction/запрос в минуту. Между запросами ставить `time.sleep(10)`.
При 429 — ждать 60 секунд до повтора.

### WebRTC на iPad — нужен HTTPS (сессия 2026-07-09)

WebRTC (используемый LiveKit для голоса/видео) НЕ работает на iPad/iOS без HTTPS.
С localhost — работает (считается secure context).
С внешнего IP/домена — требует HTTPS.

**Решение:** Caddy (автоматический SSL через Let's Encrypt, 15 минут на установку).
Или: Cloudflare Tunnel, Ngrok.

**Пока HTTPS нет — UI работает, но голос/видео не подключаются с iPad.**
