## ⚠️ КРИТИЧЕСКОЕ ПРАВИЛО: RAG-ОТРАВЛЕНИЕ — Server RAG подтягивает черновики в контекст DeepSeek (12.07.2026)

**Симптом:** При написании постов через 8005 роутер, DeepSeek получает в контекст старые черновики из `memory/`.

**Диагностика:** Если ответ содержит фразы из старых черновиков («она берёт телефон», «она пьёт кофе») — RAG подтянул мусор.

**Фикс — 3 шага:**
Шаг 1 — Очистка memory/ от черновиков. Шаг 2 — Glob-исключения в rg. Шаг 3 — writer_rag с проверенными файлами.

## ⚠️ УРОК: ПОЛНЫЙ АУДИТ ТРЕБУЕТ МНОГО ПРОХОДОВ (15.07.2026)
При инвентаризации сложной системы **один `find` никогда не даст полной картины**. Первый аудит системы для переноса упустил:
- 20+ Баз данных (показал 6, реально 29)
- Group Chat агенты (15 скриптов)
- VK боты (3 скрипта)
- Voice Assistant
- Character docs (30+ файлов)

**Правильная методика аудита (3 прохода):**
1. **Системный слой:** `systemctl list-units`, `ss -tlnp`, `docker ps`, `ps aux | grep python`
2. **Файловый слой:** `find -maxdepth 4` с правильными exclude-правилами, по несколько расширений
3. **Логический слой:** `os.walk` с подсчётом (навыки, скрипты), grep по ключевым файлам (триггеры, конфиги)

**Если результат кажется маленьким — значит что-то пропущено.** Система /opt/zinaida/ содержит 60+ директорий — один поверхностный поиск их не покроет.
