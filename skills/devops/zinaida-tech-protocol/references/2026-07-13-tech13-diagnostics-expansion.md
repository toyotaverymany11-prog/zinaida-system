# Техник 13: Расширение диагностики v2 — 13.07.2026

## Что произошло

Олег спросил: «достаточно ли этого? всю ли полностью нашу систему ты проверяешь?»

**Ответ:** Нет. Старая диагностика (7 пунктов) не проверяла:
- Docker вообще (Redis, Qdrant, n8n — а n8n жив 21 час!)
- Hermes cron jobs
- Сеть (IP, SSL, DNS)
- Кеши и .bak мусор
- Провайдеров (DeepSeek, Mistral, GitHub — реальными HTTP запросами)
- Целостность SQLite баз
- Порты наружу

## Решение

Расширен `tech_diagnostic.py` до v2 — 8 зон здоровья (зонами называются для единообразия с протоколом).

Каждая зона — отдельная функция `check_*()`, возвращающая список `(имя, ok, детали, фикс)`.

## Находки во время диагностики

### 1. n8n — жив 21 час, несмотря на все убийства
После 4 методов убийства (restart always → compose ghost → daemon resurrection → фейковый alpine-образ) контейнер всё ещё живёт.
**Причина не найдена.** Механизм воскрешения неизвестен.

### 2. Redis на 0.0.0.0:6379 — наружу
Docker `-p 6379:6379` пробросил Redis на внешний интерфейс. Любой мог слать команды.
**Фикс:** `--network host` + `redis-server --bind 127.0.0.1 --port 6379`.
После фикса: `ss -tlnp | grep 6379` → `127.0.0.1:6379`.

### 3. .bak мусор — 50+ файлов, 8.7M zinaida.bak
Hermes создаёт .bak_* файлы при каждом изменении навыка. Копились с мая.
**Фикс:** `find /root/.hermes -name '*.bak*' -exec rm -f {} +`

### 4. Dangling symlinky в systemd
`slepok.path` (not-found, но active+waiting!) и `ubuntu-fan.service` висели.
**Фикс:** `find /etc/systemd/system -lname '*/slepok.path' -delete`

### 5. Защита скрипта от ложных срабатываний

| Проблема | Решение |
|----------|---------|
| BrightData не имеет `/health` → 404 | Использовать DNS check `socket.getaddrinfo` |
| Telegram API с фейковым токеном → 404 | TCP handshake без HTTP (443) |
| docker-proxy маскирует порты | Исключать по `grep -v docker-proxy` |
| zabbix_agentd, edge_tts_server на внешних портах | Белый список известных портов |
| VK Bot на порту 5000 | Добавить в safe_ports |
