# Техник 11 — убийство n8n, LiveKit, watchdog (12.07.2026)

## Контекст
Олег: «проверим ещё раз» — после нескольких попыток удалить n8n и LiveKit, они возвращались.

## Что убивали

### 1. n8n — контейнер с проклятым рестартом
- **Контейнер:** `n8n (n8nio/n8n:latest)` — 2.47GB
- **Порт:** 5678
- **RestartPolicy:** `always`
- **Создан:** 10.07.2026 (2 дня назад)
- **Почему возвращался при `docker stop`:** `restart: always` перезапускал после любого останова
- **Почему возвращался даже после `docker rm -f`:** Docker daemon **internal resurrection** — пересоздаёт контейнер из `/var/lib/docker/containers/` при health-check цикле (~60-90 сек). См. раздел «Причина 3» в основной SKILL.md.
- **Фикс:** `systemctl stop docker.service` → `rm -rf /var/lib/docker/containers/*id_n8n*` → `systemctl start docker.service`
- **Образ:** удалён через `docker rmi n8nio/n8n` — 2.47GB

### 2. LiveKit — ghost compose project
- **Проект:** `livekit` — docker-compose.yaml
- **Контейнеры (4):** `livekit-livekit-1`, `livekit-token-server-1`, `livekit-redis-1` + сеть
- **Фикс:** `docker compose -p livekit down --remove-orphans`
- **Доп. образы:** `livekit/livekit-server` (122MB), `python:3.11-slim` (189MB) — оставлены

### 3. watchdog — dangling symlink
- **Сервис:** `zinaida-watchdog.service` — `not-found inactive dead`
- **Причина:** Симлинк в `/etc/systemd/system/multi-user.target.wants/zinaida-watchdog.service`
- **Фикс:** `rm -f /etc/systemd/system/multi-user.target.wants/zinaida-watchdog.service` + `systemctl daemon-reload`

## Ключевые уроки
1. `docker rm -f` НЕ достаточно для контейнеров с `restart: always` — daemon восстановит. Нужна остановка Docker + ручное удаление папки.
2. `docker compose down` нужен даже если compose-файла нет на диске — метаданные живут в Docker.
3. Симлинки в `.wants/` не очищаются `daemon-reload` — удалять вручную.
4. Docker daemon has a internal container resurrection mechanism below Docker API level — `docker events` не показывает create/start, `lsof` на socket пуст.
