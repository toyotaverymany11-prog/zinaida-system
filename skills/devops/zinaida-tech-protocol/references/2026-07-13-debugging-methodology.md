# Методология отладки: уроки Техник 13 (13.07.2026)

## Проблема
n8n воскресал 8 раз подряд. Я применяла docker rm -f, rmi, чистку /var/lib/docker/, alpine-заглушку — по кругу. Каждый раз один и тот же класс методов.

## Корень
Я не остановилась после 3-й неудачи, чтобы найти ИСТИННУЮ причину. Вместо этого накладывала костыли (alpine-заглушка «занять имя»), которые вели в тупик. Реальная причина — containerd snapshot #667 (1.8GB) с файловой системой n8n, который я не чистила.

## Правила (железобетонные)

### 1. Три попытки — СТОП
После 3 неудачных попыток одного класса методов — остановиться. Не применять 4-й похожий метод. Задать вопрос: «Почему это не работает? Что я не понимаю в архитектуре?»

### 2. Иди на уровень глубже
- Docker не работает? → containerd
- systemctl не помогает? → /var/lib/
- HTTP 500? → логи → код → SQL
- Не surface, а root

### 3. Хронология неудач — в файл
Создать `/opt/zinaida/memory/<problem>_history.md` ДО 4-й попытки. Записывать:
- Метод
- Результат (❌/✅/⚠️)
- Причина неудачи
- Урок

Перед новой попыткой — прочитать файл. Не повторять методы.

### 4. Не костылить мёртвое (утверждено Олегом)
Если решение — «удалить» — просто удали. Не строй guardrails, заглушки, alpine-stub. Docker сам не делает docker pull. containerd без слоя не восстановит контейнер.

Признак костыля: ты добавляешь новый компонент чтобы контролировать старый, вместо того чтобы убить старый.

| 8 | docker rm -f + rmi (без заглушки) | ✅ полностью удалён | заглушка не нужна |
