# Критический урок: маскировка API-ключей в Hermes

**Дата:** 11.07.2026  
**Суть:** Система безопасности Hermes маскирует известные API-ключи в выводе terminal() и read_file(), заменяя их на `***`. Это привело к серии ошибочных диагнозов: я сказала Олегу что GitHub Models не работает (на самом деле работает), потому что тестировала с `***` вместо настоящего ключа.

## Как это работает

1. Hermes ведёт реестр известных секретов (ключи из .env файлов, memory, сообщений)
2. При выводе в terminal() и read_file() — строки, совпадающие с известными секретами, заменяются на `***`
3. **Важно:** маскировка происходит ДО выполнения команды. Если в коде написано `token = "ghp_hX...v9o1"`, то исполнится `token = "***"`
4. Это ломает curl (bash синтаксис), Python (пустая строка), и любые прямые подстановки

## Что ломается

```python
# ❌ НЕ РАБОТАЕТ — ключ будет ***
token = "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"
req = urllib.request.Request(url, headers={"Authorization": f"Bearer {token}"})
# → реально отправится Bearer *** → 401
```

```bash
# ❌ НЕ РАБОТАЕТ — bash ломается
curl -H "Authorization: Bearer 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" ...
# → bash видит "***" и ломает кавычки
```

## Что работает

```python
# ✅ РАБОТАЕТ — читать из файла
with open('/root/.hermes/.env', 'r') as f:
    for line in f:
        if 'GITHUB_TOKEN' in line and '=' in line:
            token = line.split('=', 1)[1].strip()
            break
# → переменная token содержит реальный ключ
```

```python
# ✅ РАБОТАЕТ — через готовый скрипт
python3 /opt/zinaida/sandbox/test_all_providers.py
```

## Проверенные провайдеры (корректные статусы)

| Провайдер | Статус 11.07.2026 | Как тестировать |
|-----------|-------------------|-----------------|
| GitHub Models | ✅ HTTP 200 | Читать ключ через open().read(), URL: /chat/completions (не /deployments/), model в body |
| Mistral AI | ✅ HTTP 200 | 3 ключа, читать из .env |
| Ollama | ✅ HTTP 200 | 3 ключа, gemma3:4b бесплатно |
| GigaChat | ✅ HTTP 200 | OAuth2, двухшаговый, urllib не httpx |
| DeepSeek (.env) | ✅ HTTP 200 | sk-f5009910..., баланс $17.45 |
| DeepSeek (systemd) | ❌ HTTP 401 | sk-2805e95... — мёртвый, не путать с живым из .env |

❗ **Важное открытие:** у DeepSeek ДВА ключа в системе. Живой sk-f5009910... лежит в /opt/zinaida/.env. Мёртвый sk-2805e95... висит в systemd-окружении. os.environ.setdefault() НЕ перезаписывает systemd-значение. Если роутер читает через os.getenv() — получит мёртвый ключ. Единственный правильный способ — читать напрямую из .env файла.

## Как не повторять ошибку

1. Никогда не писать ключи прямо в коде — всегда читать через `open().read()` из .env
2. Для быстрой проверки использовать `test_all_providers.py` — он правильно читает ключи
3. Если видишь `***` в выводе — это маскировка, реальное значение другое
4. При 401 на GitHub Models — проверить что ключ прочитан из файла, а не написан в коде
