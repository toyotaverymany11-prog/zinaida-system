# Ошибка 500 с чистыми логами — версия пакета vs кэш браузера

## Симптом
- Пользователь загружает .docx файл через интерфейс Hermes Studio
- UI выдаёт ошибку 500
- Агент перестаёт отвечать ("вылетает")
- В server.log **нет** сообщений об ошибках — только `level:30` (info)
- В journalctl **нет** записей об ошибках — только `Failed with result 'timeout'` (может быть, может не быть)

## Контекст
- Hermes Studio установлен глобально через npm
- `npm list -g hermes-web-ui` показывает 0.6.10
- `npm view hermes-web-ui version` показывает 0.6.27 (доступно обновление)
- server.log показывает `Update available: 0.6.10 → 0.6.27` каждые несколько секунд

## Корневая причина
Пользователь или процесс уже установил новую версию (`npm update -g hermes-web-ui` и `systemctl restart hermes-web-ui`), но браузер на клиенте кэшировал старый JS-бандл (0.6.10). Когда браузер пытается отправить запрос по старому API, сервер (уже 0.6.27) не понимает формат → 500.

Логи сервера при этом чистые, потому что сервер корректно обрабатывает запрос — он просто возвращает 500 на неправильный формат данных. Ошибка на уровне HTTP, не на уровне приложения.

## Фикс
1. **Ctrl+Shift+R** (жёсткая перезагрузка без кэша) в браузере
2. Если не помогло — очистить кэш браузера полностью:
   - Chrome/Firefox: Настройки → Конфиденциальность → Очистить кэш
3. Открыть вкладку в режиме инкогнито (гарантированно без кэша)

## Профилактика
После любого обновления Hermes Studio:
```bash
npm update -g hermes-web-ui
systemctl restart hermes-web-ui
# Сообщить пользователю: сделать Ctrl+Shift+R в браузере
```

## Запись в memory для себя
При анализе ошибки 500 в Hermes Studio:
1. Всегда проверять `npm view hermes-web-ui version` vs `npm list -g hermes-web-ui`
2. Если версии не совпадают — первым делом сказать про кэш браузера
3. Не тратить время на глубокое копание логов, если версии разные
