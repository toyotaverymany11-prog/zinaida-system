# Трассировка происхождения сессии в Hermes Studio

**Когда:** Пользователь спрашивает «кто создал этот чат?», «откуда взялась эта сессия?»,  
«агенты сами создают чаты — разберись».

**Проблема:** В списке сессий Hermes Studio появляются сессии, которые пользователь НЕ создавал.  
Они выглядят как обычные чаты, но с непонятным первым сообщением (часто тестовым вроде «Say hello in 3 words»).

---

## Коренная причина

Hermes Agent при определённых условиях создаёт тестовые сессии автоматически:

1. **Переключение модели** (`switch_model`) — когда агент переключается между провайдерами или моделями
2. **Восстановление MCP-серверов** — после падения/перезапуска MCP соединений
3. **MainThread инициализация** — `agent_init` или `switch_model` на главном потоке

**Паттерн авто-созданной сессии:**

| Признак | Значение |
|---------|----------|
| source | `cli` |
| agent | `hermes` |
| platform | `cli` |
| thread | `MainThread` |
| history | `0` (новая сессия, без истории) |
| Первое сообщение | «Say hello in 3 words» или другое тестовое |
| parent_session_id | `null` |
| Модель при создании | Может отличаться от основной (mistral, zinaida-router) |

---

## Протокол диагностики

### Шаг 1: Получить метаданные сессии

Через Hermes Studio API:

```
# MCP: mcp_hermes_studio_use_hermes_studio_use_session_get
# или mcp_hermes_studio_use_hermes_studio_use_sessions_list
```

**Ключевые поля:**
- `source` — `cli` / `cron` / `studio`
- `agent` — `hermes` / `""` (пусто = через Web UI)
- `platform` — `cli` / `cron` / `webui`
- `started_at` — Unix timestamp создания
- `parent_session_id` — если не null, это дочерняя сессия от другой
- `model` — какая модель использовалась при создании
- `history` (в логах) — количество сообщений в сессии на момент первого обращения

### Шаг 2: Проверить кто был онлайн в момент создания

```bash
# SSH сессии
last | head -20

# Сопоставить IP с временем создания сессии (started_at → человеческое время)
date -d @<timestamp> '+%Y-%m-%d %H:%M:%S'
```

Если в момент создания сессии никто не был залогинен — сессия авто-созданная.

### Шаг 3: Найти первое упоминание сессии в логах агента

```bash
# agent.log (текущий) + agent.log.1 (ротированный — может хранить исторические данные)
grep "<session-id>" /root/.hermes/logs/agent.log /root/.hermes/logs/agent.log.1 2>/dev/null
```

**Ключевые строки:**
```
agent.turn_context: ... platform=cli history=0 msg='Say hello in 3 words'
```
- `platform=cli` — CLI канал
- `history=0` — первое сообщение в новой сессии
- `thread=MainThread` — создано главным потоком (автоматика)
- `thread=hermes-bridge-...` — создано через bridge (пользовательский запрос)

**Дополнительные маркеры:**
```
OpenAI client created (agent_init, shared=True) thread=MainThread
OpenAI client created (switch_model, shared=True) thread=MainThread
Model switched in-place: ...
```
Если эти строки непосредственно ПЕРЕД созданием сессии — это авто-создание при переключении модели.

### Шаг 4: Проверить что происходило в системе в тот момент

```bash
# Логи агента за ±2 минуты от создания сессии
sed -n '/<время-2мин>/,/<время+2мин>/p' /root/.hermes/logs/agent.log.1
```

**Типовые спутники авто-создания:**
- `OAuth flow error` — MCP сервер не смог подключиться
- `MCP server 'linear' initial connection failed` — MCP падение
- `cron.scheduler: Job '...': 41 MCP tool(s) available` — запуск cron после восстановления
- `Failed to connect to MCP server` — множественные ошибки MCP
- `environment_probe` — если включено (config.yaml: `environment_probe: true`)

### Шаг 5: Проверить request_dump (если есть)

```bash
ls -la /root/.hermes/sessions/ | grep <session-id>
```

В дампе видно какой system prompt использовался — если это полный профиль Зинаиды, значит сессия была создана через обычный агентский флоу, не через внешний API.

## Типовые сценарии

### Сценарий A: Авто-созданная при переключении модели

**Симптомы:**
- `platform=cli`, `thread=MainThread`
- В логах перед созданием: `switch_model`, `agent_init`
- Пользователь не был онлайн
- Первое сообщение: «Say hello in 3 words»

**Решение:** Это внутреннее поведение Hermes Agent. 
- Если бесит — можно отключить `environment_probe: false` в config.yaml  
  (но это может скрыть другие проблемы)
- Либо задокументировать как известное поведение и не паниковать

### Сценарий B: Созданная через Hermes Studio Web UI

**Симптомы:**
- `source=studio` (или `source=""` с `agent=""`)
- Пользователь был онлайн в это время
- Первое сообщение — осмысленное (не тестовое)

**Решение:** Нормальное поведение. Пользователь создал сессию сам.

### Сценарий C: Созданная через cron job

**Симптомы:**
- `platform=cron`
- id сессии начинается с `cron_`
- Есть job_id в логах

**Решение:** Проверить cron jobs.json → найти job → прочитать его описание.

### Сценарий D: Созданная через delegate_task / суб-агента

**Симптомы:**
- Есть `parent_session_id`
- В логах есть `hermes_studio_use_chat_run`
- Сессия создана в то же время когда работал суб-агент

**Решение:** Исправить подход — суб-агенты не должны использовать `hermes_studio_use_chat_run`.

---

## Профилактика

Чтобы авто-созданные сессии не мозолили глаза:
1. Задокументировать как известное поведение Hermes Agent
2. Научить команду отличать (source=cli + MainThread = авто)
3. Можно настроить авто-архивацию/удаление пустых сессий через cron

**Не рекомендуется:** отключать `environment_probe` — это диагностический инструмент, который помогает обнаруживать проблемы с моделями при старте.
