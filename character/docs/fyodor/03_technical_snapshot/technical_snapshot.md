# Technical Snapshot: как система работает внутри

Этот документ написан для маркетолога, поэтому без глубокого кода. Главная задача - понимать, что за красивыми словами стоит реальная система, а не один промпт.

## Простая схема

Пользователь пишет Зинаиде в Telegram или на сайте.

Система:

1. Определяет пользователя.
2. Проверяет баланс токенов, тариф, промокоды и доступ к функции.
3. Понимает тип сообщения: обычный чат, отношения, тест, Лаборатория, голос, видео, платеж.
4. Поднимает память: факты, эпизоды, людей, предыдущие ситуации.
5. При необходимости поднимает знания из RAG.
6. Собирает промпт для модели.
7. Отправляет задачу в очередь/worker, если генерация тяжелая.
8. Получает ответ от LLM или другого провайдера.
9. Постобрабатывает ответ: тон, длина, safety, кнопки "дальше".
10. Сохраняет полезное в память и историю.
11. Отправляет ответ пользователю.

## Основные технические блоки

**Bot.** Telegram-часть. Принимает сообщения, показывает меню, запускает функции, отправляет ответы.

**Web.** Сайт с авторизацией, чатом, Лабораторией, тестами, оплатой, промокодами и вкладкой звонка.

**Postgres.** Основная база данных. Нужна для пользователей, памяти, платежей, задач, истории, токенов.

**Redis.** Быстрая очередь и слой для лимитов. Помогает не перегружать систему.

**Worker.** Отдельный процесс для генераций. Он берет задачи из очереди, делает тяжелую работу и возвращает результат.

**Payment watcher.** Следит за платежами, проверяет статусы и начисляет токены.

**Admin.** Внутренняя панель: платежи, RAG, память, системные проверки.

**RAG/Knowledge.** База знаний и механизм поиска по ней.

**Prompt layer.** Слой, который собирает стиль, память, safety, знания и конкретную задачу в один запрос к модели.

## Карточки технических блоков

Эти карточки можно использовать в презентации для маркетолога, инвестора или подрядчика. Они объясняют не код, а роль каждого блока в продукте.

### 1. Telegram Bot

**Роль.** Быстрый вход в Зинаиду через привычный мессенджер.

**С чем связан.** Пользователь, платежи, токены, память, Лаборатория, worker, Postgres, Redis.

**Что дает продукту.** Минимальный барьер входа: пользователь просто пишет сообщение.

**Что важно для маркетинга.** Telegram хорош для beta, первых продаж, retention и быстрых касаний.

**Ограничение.** Telegram слабее сайта по визуалу, личному кабинету и сложной навигации.

### 2. Web-сайт

**Роль.** Основная будущая витрина и полноценный интерфейс Зинаиды.

**С чем связан.** Авторизация, чат, Лаборатория, тесты, платежи, промокоды, история, звонок, общая база.

**Что дает продукту.** Красивый опыт, личный кабинет, понятную оплату, историю разборов и более дорогую упаковку.

**Что важно для маркетинга.** Сайт проще показывать в рекламе, презентациях и лендингах.

**Ограничение.** На сайте пользователь ожидает polish: дизайн, скорость, понятные состояния, аккуратные ошибки.

### 3. Авторизация и профиль пользователя

**Роль.** Понимать, кто перед нами, какой у него баланс, история и доступы.

**С чем связан.** Telegram ID, web-аккаунт, Postgres, платежи, промокоды, память.

**Что дает продукту.** Одна история пользователя между каналами.

**Что важно для маркетинга.** Можно вести пользователя из Telegram на сайт и обратно, не теряя ценность.

**Ограничение.** Нужно аккуратно решать связку аккаунтов, чтобы не было дублей и потери баланса.

### 4. Postgres

**Роль.** Главная база данных.

**С чем связан.** Пользователи, платежи, токены, память, эпизоды, задачи, история, промокоды.

**Что дает продукту.** Надежное хранение данных и переход от beta-локальности к нормальной архитектуре.

**Что важно для маркетинга.** Можно честно говорить, что у продукта есть база, история и управляемая память, а не временные сообщения.

**Ограничение.** Нужны backup, миграции, мониторинг и аккуратная работа с персональными данными.

### 5. Redis

**Роль.** Быстрый слой для очередей, лимитов и временных состояний.

**С чем связан.** Worker, bot, web, лимиты, backpressure.

**Что дает продукту.** Помогает переживать пики нагрузки и не пытаться делать все одновременно.

**Что важно для маркетинга.** Позволяет масштабировать запуск аккуратнее: система может ставить задачи в очередь.

**Ограничение.** Redis не заменяет основную базу и требует правильных правил восстановления.

### 6. Worker

**Роль.** Отдельный исполнитель тяжелых задач.

**С чем связан.** Redis, Postgres, LLM-провайдеры, bot, web, payment watcher.

**Что дает продукту.** Бот и сайт не зависают, пока модель генерирует длинный ответ, голос или видео.

**Что важно для маркетинга.** Можно запускать больше пользователей, не превращая каждый ответ в блокировку интерфейса.

**Ограничение.** Если worker перегружен, ответы будут приходить с задержкой. Нужны лимиты и мониторинг очереди.

### 7. LLM-провайдер

**Роль.** Генерирует текстовые ответы Зинаиды.

**С чем связан.** Prompt layer, память, RAG, safety, worker.

**Что дает продукту.** Собственно умное общение, разборы, тесты и сценарии.

**Что важно для маркетинга.** Качество Зинаиды зависит не только от модели, но и от памяти, промптов, UX и постобработки.

**Ограничение.** Провайдер может иметь лимиты, задержки, ошибки и переменную стоимость.

### 8. Prompt layer

**Роль.** Собирает "мозг ответа": стиль Зинаиды, задачу, память, safety, RAG и ограничения по длине.

**С чем связан.** LLM, analyzer, память, RAG, Лаборатория, safety.

**Что дает продукту.** Делает ответы не случайными, а управляемыми: короче, теплее, точнее по сценарию.

**Что важно для маркетинга.** Именно здесь рождается характер Зинаиды.

**Ограничение.** Если правил слишком много, они начинают конкурировать. Нужна регулярная чистка и тесты диалогов.

### 9. Memory layer

**Роль.** Хранит важные факты, эпизоды, людей и паттерны.

**С чем связан.** Чат, Лаборатория, Postgres, analyzer, prompt layer, admin.

**Что дает продукту.** Эффект "она меня помнит".

**Что важно для маркетинга.** Это одна из главных причин возвращаться.

**Ограничение.** Память должна быть управляемой: показать, исправить, удалить.

### 10. RAG/Knowledge layer

**Роль.** Достает знания из внутренней базы документов.

**С чем связан.** Prompt layer, admin, база документов, embeddings, чат, Лаборатория.

**Что дает продукту.** Ответы могут быть более содержательными, особенно в темах отношений, коммуникации и самоанализа.

**Что важно для маркетинга.** Это будущая основа "экспертного мозга".

**Ограничение.** Пока это beta-слой. Для сильного claim нужны источники, цитирование, паспорта документов и контроль качества.

### 11. Relationship Lab engine

**Роль.** Запускает структурные сценарии Лаборатории.

**С чем связан.** Чат, память, токены, prompt layer, Postgres, UI-кнопки.

**Что дает продукту.** Превращает разговор в понятный инструмент с результатом.

**Что важно для маркетинга.** Лаборатория - самый понятный платный модуль.

**Ограничение.** Нельзя превращать разбор в диагноз или приговор другому человеку.

### 12. Billing и токены

**Роль.** Считает доступы, списания, тарифы и остаток.

**С чем связан.** ЮKassa, промокоды, Postgres, bot, web, Лаборатория, голос, видео.

**Что дает продукту.** Коммерческую модель: пользователь платит за реальные действия.

**Что важно для маркетинга.** Можно упаковывать функции по ценности, а не продавать "безлимитный AI".

**Ограничение.** Нужно очень понятно показывать, за что списались токены.

### 13. YooKassa/payment watcher

**Роль.** Принимает оплату и проверяет статус платежей.

**С чем связан.** Billing, Postgres, web, Telegram, тарифы, токен-паки.

**Что дает продукту.** Деньги, карта/СБП, пополнение токенов.

**Что важно для маркетинга.** Можно продавать в рублях и запускать понятные офферы.

**Ограничение.** Нужны проверка webhook/статусов, правила возвратов, оферта и бухгалтерия.

### 14. Promocodes

**Роль.** Дает скидки, бонусы или trial-доступ.

**С чем связан.** Billing, Postgres, web, Telegram, маркетинговые каналы.

**Что дает продукту.** Инструмент для блогеров, акций, возврата и экспериментов.

**Что важно для маркетинга.** Можно измерять каналы и делать разные офферы.

**Ограничение.** Нужны лимиты, срок действия и защита от злоупотреблений.

### 15. Voice/Video providers

**Роль.** Превращают текстовую Зинаиду в голос или видео.

**С чем связан.** Worker, billing, лимиты, prompt layer, bot, web.

**Что дает продукту.** Эмоциональный premium-слой.

**Что важно для маркетинга.** Это сильный wow-материал для демонстрации.

**Ограничение.** Дорого, медленнее текста, требует жестких лимитов.

### 16. Admin и monitoring

**Роль.** Дает команде контроль над системой.

**С чем связан.** RAG, память, платежи, очереди, ошибки, Postgres.

**Что дает продукту.** Операционную зрелость: можно видеть проблемы, а не узнавать о них только от пользователя.

**Что важно для маркетинга.** Перед платным трафиком команда должна видеть конверсии, ошибки платежей, задержки и падения.

**Ограничение.** Нужны alerts и продуктовая аналитика, не только техническая админка.

## Как блоки связаны между собой

### Поток обычного сообщения

Пользователь -> Telegram/сайт -> профиль -> проверка токенов -> analyzer -> память -> RAG при необходимости -> prompt layer -> worker/LLM -> постобработка -> сохранение истории -> ответ пользователю.

**Смысл для маркетинга.** Даже обычный ответ - это не один запрос в модель. Это цепочка, где память и стиль делают Зинаиду узнаваемой.

### Поток Лаборатории

Пользователь выбирает инструмент -> billing проверяет токены -> Лаборатория собирает входные данные -> память поднимает историю -> prompt layer задает формат -> LLM делает разбор -> результат сохраняется как эпизод/артефакт -> пользователь получает кнопки "дальше".

**Смысл для маркетинга.** Лаборатория должна продаваться как действие с продолжением, а не как разовый текст.

### Поток оплаты

Пользователь выбирает тариф -> web/Telegram создает платеж -> ЮKassa принимает оплату -> payment watcher проверяет статус -> Postgres фиксирует платеж -> billing начисляет токены -> пользователь получает доступ.

**Смысл для маркетинга.** Важно объяснять пользователю, что он получил после оплаты: токены, срок, функции, остаток.

### Поток памяти

Пользователь рассказывает историю -> analyzer выделяет важное -> система сохраняет факт/эпизод/человека -> следующие ответы поднимают релевантное -> пользователь видит продолжение.

**Смысл для маркетинга.** Память превращает Зинаиду из "сервиса ответа" в "отношение с продуктом".

### Поток RAG

Пользователь задает вопрос -> analyzer понимает, что нужны знания -> RAG ищет документы -> prompt layer добавляет найденное -> LLM отвечает с учетом базы -> при развитии можно показывать источники.

**Смысл для маркетинга.** RAG - не отдельная кнопка, а усилитель качества. Его можно продавать сильнее только после усиления источников.

### Поток голоса/видео

Пользователь выбирает голос/видео -> billing проверяет токены -> LLM готовит текст -> voice/video provider генерирует медиа -> worker ждет результат -> ответ отправляется пользователю -> списание и история фиксируются.

**Смысл для маркетинга.** Голос/видео должны ощущаться как premium-магия, но технически это дорогой и медленный слой.

### Поток нагрузки

Много пользователей пишут одновременно -> bot/web быстро принимают запросы -> задачи уходят в Redis/очередь -> worker обрабатывает с лимитом -> Postgres хранит состояние -> при перегрузе включается backpressure.

**Смысл для маркетинга.** Масштабирование - это не "дать сервер побольше", а управлять очередью, лимитами, провайдерами и ожиданиями пользователя.

## Простая метафора системы

Если объяснять совсем коротко:

- Telegram и сайт - двери.
- Авторизация - паспорт пользователя.
- Billing - касса и счетчик токенов.
- Память - личная история пользователя.
- RAG - библиотека.
- Prompt layer - режиссер ответа.
- LLM - генератор речи и смысла.
- Лаборатория - набор рабочих инструментов.
- Worker - производственный цех.
- Postgres - архив и бухгалтерская книга.
- Redis - очередь на кухне.
- Admin - пульт управления.

Эта метафора полезна для презентаций: становится видно, что Зинаида - это система, а не один чат.

## Почему это важно для маркетинга

Зинаиду можно честно описывать как систему с несколькими слоями:

- личность и стиль;
- память;
- инструменты Лаборатории;
- база знаний;
- биллинг;
- сайт и Telegram;
- техническая очередь для масштабирования.

Это сильнее, чем "мы подключили ChatGPT к боту".

## Что уже улучшено под нагрузку

Система была переведена в сторону нормальной production-архитектуры:

- вместо простого локального хранения используется Postgres;
- тяжелые генерации выносятся в worker;
- Redis используется для очередей и лимитов;
- есть durable queue, чтобы задачи не терялись при обычной нагрузке;
- есть управляемые лимиты для текста, голоса и видео;
- проводились synthetic load tests на сценарии 500 активных пользователей.

## Где реальное бутылочное горлышко

Главный риск при росте - не сама база данных, а внешние AI-провайдеры:

- лимиты запросов;
- скорость ответа модели;
- стоимость генераций;
- отказы или задержки API.

Поэтому маркетинговый запуск должен идти вместе с лимитами, мониторингом и очередями. Нельзя просто резко залить большой трафик без staged rollout.

## Что нельзя обещать технически

Нельзя обещать:

- мгновенные ответы всегда и всем;
- 100% отсутствие ошибок провайдера;
- бесконечные голосовые и видео без лимитов;
- полноценную медицинскую/психологическую безопасность только за счет AI;
- идеально точную память без права пользователя исправлять и удалять ее.
